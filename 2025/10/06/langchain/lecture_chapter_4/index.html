4. RAG의 모든 것 (Retrieval-Augmented Generation)

작성일 2025-10-06

🎯 이 챕터에서 배울 것

  • RAG(검색 증강 생성)의 기본 개념과 전체 흐름(Load, Split, Embed, Store, Retrieve) 이해하기
  • UnstructuredFileLoader를 사용하여 다양한 형식의 문서를 로드하는 방법
  • CharacterTextSplitter와 Tiktoken을 사용하여 문서를 의미 있는 조각(chunk)으로 분할하는 방법
  • OpenAIEmbeddings와 VectorStore(FAISS/Chroma)를 사용하여 문서 조각을 벡터로 변환하고 저장하는 방법
  • RetrievalQA 체인을 사용하여 RAG 파이프라인을 간단하게 구축하는 방법
  • LCEL을 사용하여 Stuff 및 Map-Reduce와 같은 RAG 체인을 직접 구현하는 방법

Load & Split (feat. Tiktoken)

🎯 이번 단계에서 배울 것

  • RAG의 첫 두 단계인 **Load(로드)**와 **Split(분할)**의 개념을 이해합니다.
  • UnstructuredFileLoader를 사용하여 .docx, .txt 등 다양한 파일을 로드합니다.
  • CharacterTextSplitter를 사용하여 로드된 문서를 작은 조각으로 분할합니다.
  • from_tiktoken_encoder를 사용하여 OpenAI 모델의 토큰 계산 방식에 맞춰 더 정확하게 분할합니다.

🔍 코드 상세 설명

1. RAG의 첫걸음: Load & Split
LLM은 한 번에 처리할 수 있는 텍스트의 양(컨텍스트 윈도우)에 제한이 있습니다. 따라서 긴 문서를 LLM에게 제공하려면, 먼저 문서를 불러와(Load) 잘게 쪼개는(Split) 과정이 필요합니다.

  • UnstructuredFileLoader: 텍스트, PDF, 워드 문서 등 다양한 형식의 파일을 처리할 수 있는 강력한 로더입니다.
  • CharacterTextSplitter: 가장 기본적인 스플리터로, 지정된 문자를 기준으로 텍스트를 분할합니다.
  • from_tiktoken_encoder: 단순히 글자 수가 아닌, OpenAI의 Tiktoken 라이브러리를 사용하여 토큰 수를 기준으로 chunk_size를 계산합니다. 이는 모델의 컨텍스트 윈도우를 더 정확하게 관리할 수 있게 해줍니다.
  • chunk_overlap: 분할된 조각들이 서로 약간의 내용을 겹치게 만들어, 문맥이 잘리는 것을 방지하고 의미가 잘 유지되도록 돕습니다.

  • load_and_split 메소드로 문서 로드와 분할을 한 번에 실행했나요?

Embed & Store (Vector Stores)

🎯 이번 단계에서 배울 것

  • RAG의 세 번째와 네 번째 단계인 **Embed(임베딩)**와 **Store(저장)**의 개념을 이해합니다.
  • OpenAIEmbeddings를 사용하여 텍스트 조각을 벡터(숫자의 배열)로 변환합니다.
  • FAISS 또는 Chroma와 같은 벡터 스토어에 임베딩된 벡터를 저장하고 검색하는 방법을 배웁니다.
  • CacheBackedEmbeddings를 사용하여 임베딩 과정을 캐싱하고 비용을 절약합니다.

전체 코드 (notebook.ipynb):

🔍 코드 상세 설명

1. 임베딩 (Embedding)
임베딩은 텍스트를 컴퓨터가 이해할 수 있는 숫자의 배열, 즉 벡터(vector)로 변환하는 과정입니다. 이 벡터는 텍스트의 의미를 다차원 공간의 한 점으로 표현합니다. 의미가 비슷한 텍스트들은 공간상에서 가까운 위치에 있게 됩니다.

  • OpenAIEmbeddings: OpenAI의 임베딩 모델을 사용하여 텍스트를 벡터로 변환합니다.

2. 벡터 스토어 (Vector Store)
벡터 스토어는 생성된 벡터들을 효율적으로 저장하고, 특정 벡터와 의미적으로 가장 유사한 다른 벡터들을 빠르게 검색할 수 있는 데이터베이스입니다.

  • FAISS / Chroma: 대표적인 오픈소스 벡터 스토어 라이브러리입니다.
  • from_documents: 문서 조각(docs)과 임베딩 모델(cached_embeddings)을 받아, 내부적으로 각 조각을 벡터로 변환한 후 벡터 스토어에 저장합니다.
  • similarity_search: 주어진 질문을 벡터로 변환한 뒤, 벡터 스토어에서 가장 가까운(유사한) 벡터들을 가진 문서 조각들을 찾아 반환합니다.

3. CacheBackedEmbeddings
임베딩은 비용이 발생하는 API 호출일 수 있습니다. CacheBackedEmbeddings는 한 번 임베딩된 결과를 파일 시스템(LocalFileStore)에 저장하여, 동일한 텍스트에 대한 임베딩 요청이 다시 들어올 경우 API를 호출하는 대신 캐시에서 결과를 가져와 비용과 시간을 절약합니다.

  • RetrievalQA 체인을 사용하여 RAG 파이프라인을 쉽게 구축하는 방법

전체 코드 (notebook.ipynb):

🔍 코드 상세 설명

1. RetrievalQA
RetrievalQA는 RAG의 전체 과정을 추상화하여 제공하는 편리한 체인입니다. 내부적으로 다음과 같은 작업을 수행합니다.

  1. 사용자의 질문을 받습니다.
  2. retriever를 사용하여 질문과 관련된 문서 조각들을 벡터 스토어에서 검색합니다.
  3. 검색된 문서 조각들을 질문과 함께 프롬프트에 넣어 LLM에게 전달합니다.
  4. LLM은 주어진 문서 조각(context)을 기반으로 질문에 대한 답변을 생성합니다.
  • retriever: vectorstore.as_retriever()를 통해 벡터 스토어를 검색기 인터페이스로 변환합니다.
  • chain_type: 검색된 문서들을 처리하는 방식을 결정합니다. (자세한 내용은 다음 섹션에서 다룹니다.)

  • chain.run으로 질문을 하고, 문서 내용을 기반으로 한 답변을 받았나요?

LCEL 기반 RAG 체인 (Stuff & Map-Reduce)

🎯 이번 단계에서 배울 것

  • LCEL을 사용하여 RAG 파이프라인을 직접 제어하고 커스터마이징하는 방법
  • Stuff: 가장 간단한 RAG 체인으로, 검색된 모든 문서를 하나의 프롬프트에 넣는 방식
  • Map-Reduce: 문서가 너무 많아 하나의 컨텍스트에 다 들어가지 않을 때 사용하는 고급 기법

전체 코드 (notebook.ipynb):

🔍 코드 상세 설명

1. Stuff Chain

  • retriever가 검색한 모든 문서 조각을 {context} 플레이스홀더에 “욱여넣는(stuff)” 가장 직관적인 방식입니다.
  • 간단하고 성능이 좋지만, 검색된 문서의 총량이 모델의 컨텍스트 윈도우를 초과하면 오류가 발생합니다.

2. Map-Reduce Chain

  • Map 단계: 검색된 각 문서 조각에 대해 독립적으로 질문을 던져 중간 요약/답변을 생성합니다. (map_doc_chain)
  • Reduce 단계: 생성된 모든 중간 답변들을 모아, 최종적으로 하나의 종합적인 답변을 생성합니다. (final_prompt | llm)
  • 문서의 양이 매우 많을 때 유용하지만, 여러 번의 LLM 호출로 인해 비용이 더 많이 들고 시간이 오래 걸릴 수 있습니다.
